### What is DNS?

DNS, or a domain name system, is used to connect domain names with IP addresses. The reason why it exists is because humans aren't very good at remembering long strings of numbers, while machines only understand strings of zeroes and ones. Prior to the existence of DNS we had a hosts file, during the time of ARPAnet. 

Through the 1970s, the ARPAnet was a small community of a few hundred hosts. A single file, HOSTS.TXT, contained a name-to-address mapping. Whenever a new host would be added to the network, administrators of said host would email hosts.txt file administrators in order for that device to be added, and then periodically FTP’ed to the machine which contained hosts file and grabbed the latest version. Their changes were compiled into a new HOSTS.TXT file once or twice a week. However, with the exponential growth of the network list of issues with HOSTS.TXT file became more evident:

- Traffic and load: The more hosts were added to the network, the more of them attempted to connect to the SRI-NIC to distribute the HOSTS.TXT file.
- Name collisions: Although NIC could control uniqueness of addresses, it couldn't control the hostname. If two hostnames shared the same name, it would cause conflicts across the entire network - for example, it could disrupt mail ss mapping for every host connected to the ARPAnet. HOSTS.TXT was maintained by SRI’s Network Informaervice.
- Consistency: The changes being made to the HOSTS.TXT file couldn't keep up with an expanding network. By the time HOSTS.TXT file reached the furthest portions of the network, new hosts would already be added to it and it would have to be updated again.

New system was required, one which would allow local administration of data yet make that data globally available. 
- The decentralization of administration would eliminate the single-host bottleneck and relieve the traffic problem. 
- Local management would make the task of keeping data up-to-date much easier. The new system should use a hierarchical namespace to name hosts. This would ensure the uniqueness of names. 
- Making domain names hierarchical eliminates the problem of name collisions. Each domain has a unique domain name, so the organization that runs the domain is free to name hosts and subdomains within its domain. Whatever name they choose for a host or subdomain won’t conflict with other organizations’ domain names because it will end in their unique domain name.

The idea was to structure DNS to be like the Unix filesystem - it's an inverted tree with the root node at the top and each additional node as a branch on the tree. Each of those nodes is what we call domains. 

Domains can be separated by the level they are on:

- A first level domain/**top level domain** is a child of the root
- A second-level domain is a child of a first-level domain, and so on

Domains can also be separated by their positioning:

- Relative position is represented in relation to the node's parent note.
- Absolute position is represented in relation to the root node.

This positioning is equivalent to UNIX filesystem's relative and absolute paths. 

Absolute path of the domain has a dot at the end of the domain name which represents root node and it's also called a **fully qualified domain name**, or in short **FQDN**.
`www.google.com.`

Segments of the overall system are controlled locally but data in each segment is available across the entire network through a **client/server scheme** where nameservers constitute a server portion, while resolvers constitute the client portion. 

**Nameservers** are programs which store complete information about some segments of the system called a DNS zone, which they load from a file or from another nameserver. They will have full authority of that zone and they make that information available to **resolvers**, which are library routines that create queries and send them across a network to a nameserver.

##### Types of Nameservers (extra read, not required)

The DNS specs define two types of nameservers: 
- **primary**: A primary nameserver reads the data for the zone from a file on its host. 
- **secondary**: A secondary nameserver (also known as a slave nameserver) gets the zone data from another nameserver authoritative for the zone, called its master server.

**Primary DNS server**  [DNS server](https://www.cloudflare.com/learning/dns/dns-server-types/) is a server that hosts a website’s primary [zone file](https://www.cloudflare.com/learning/dns/glossary/dns-zone/). This is a text database file that contains all of the authoritative information for a domain, including its IP address, the identity of the domain administrator, and various resource records. Primary servers are also responsible for making any necessary changes to a zone’s [DNS records](https://www.cloudflare.com/learning/dns/dns-records/). Once the primary server has completed the update, it can then pass along change requests to the secondary servers.

**Secondary DNS server** - Primary DNS servers contain all relevant resource records and handle DNS queries for a [domain](https://www.cloudflare.com/learning/dns/glossary/what-is-a-domain-name/). By contrast, secondary DNS servers contain zone file copies that are read-only, meaning they cannot be modified. Instead of getting their information from local files, they receive pertinent information from a primary server in a communication process known as a zone transfer.

Although each zone is limited to one primary DNS server, it can have any number of secondary DNS servers. Maintaining one or more secondary servers ensures that queries can be resolved even if the primary server becomes unresponsive.

There are two main benefits of using a secondary DNS server:

- **Redundancy and resiliency:** Relying on just one DNS server creates a single point of failure. If the primary server fails or is compromised by an [attack](https://www.cloudflare.com/learning/ddos/what-is-a-ddos-attack/), prospective visitors can no longer access the desired domain. Using secondary servers creates redundancy and makes it less likely that users will experience a disruption of service.
- **Load balancing:** Secondary DNS servers can share the burden of incoming requests to the domain so that the primary server doesn’t get overloaded and cause a denial-of-service. They do this using [round-robin DNS](https://www.cloudflare.com/learning/dns/glossary/round-robin-dns/), a load balancing technique designed to send roughly equal amounts of traffic to each server.****

DNS provides two types of nameservers to make administration easier. Once the data for the zone has been created and the primary nameserver set up, it is unnecessary to copy all that data from host to host to create new nameservers for the zone. Instead, simply setting up the slave nameservers to load their data from the primary master of the zone is enough, as they will transfer the new zone data when necessary.

**Zone datafiles** are files from which primary master nameservers load their zone data.  The datafiles contain **resource records** that describe the zone. Slave nameservers can also load their zone data from datafiles, and they are are usually configured to back up the zone data they transfer from a master nameserver to datafiles. If the slave is later killed and restarted, it reads the backup datafiles first, then checks to see whether its zone data is current. ****

**Resolvers** are the clients that access nameservers, and programs running on a host that need information from the domain namespace use the resolver to get them. The resolver handles:

- Querying a nameserver
- Interpreting responses (which may be resource records or an error)
- Returning the information to the programs that requested it

The resolver relies almost entirely on the nameservers it queries: it has the smarts to put together a query, to send it and wait for an answer, and to resend the query if it isn’t answered, but the burden of finding an answer to the query is placed on the nameserver. The DNS specs call this kind of resolver a **stub resolver**.

### How does a recursive DNS resolution process occur?

When a DNS query is initiated, stub resolver will be asked by the program to check the system caches first to see if the domain name and its corresponding IP address are already stored there. It will visit caches in the following order:
   - Browser cache
   - OS cache
   - Hosts file
   - Router cache
   - ISP cache

If the DNS zone files aren't cached, [stub resolver](https://www.nslookup.io/learning/what-is-a-dns-resolver/) will then convert the name resolution requests from applications like web browsers into DNS request messages. 

Stub resolvers are aware only of your local resolvers which are configured in the /etc/resolv.conf file and the stub resolver sends the DNS request messages to them. It'll ask if they know where to find the domain `touchsupport.com`, and at that point they'll take the request and start a recursive query from the first location they are aware of - root nameservers.

**2. TLD (Top Level Domain) Nameservers**: TLD nameservers store information for domain names that share a common domain extension. For example, a .com TLD nameserver contains information for all websites with a ".com" extension. TLD nameservers can be categorized into two main groups:
  - Generic top-level domains (gTLDs): These are domain extensions that are not country-specific, such as .com, .org, .net, .edu, and .gov.
  - Country code top-level domains (ccTLDs): These include domain extensions specific to a country or state, such as .rs for Serbia.

After receiving a response from a root nameserver, the recursive resolver sends a query to the corresponding TLD nameserver for the requested domain. The TLD nameserver does not have access to the A record of the domain we need but it is aware of which authority manages the DNS zone for that domain. TLD-nameserver then points the recursive resolver toward the domain registrar that holds the information for authoritative nameservers where DNS zone is being hosted.

**4. Authoritative Nameservers**: Once the recursive resolver receives a response from the TLD nameserver, it is directed to an authoritative nameserver. The authoritative nameserver holds domain-specific information and can provide the recursive resolver with the IP address of the server associated with the domain (found in the DNS A record). If the authoritative nameserver contains an alias domain (CNAME record) for the requested domain, it provides the recursive resolver with the alias domain. In such cases, the recursive resolver would need to perform a new DNS lookup to obtain the necessary records from the authoritative nameserver.
```
touchsupport.com.	14400	IN	A	184.106.55.138
```

**Summary:** A stub resolver queries a recursive resolver, which then sends queries to other nameservers in pursuit of an answer for the resolver. It will visit root nameserver ->TLD nameserver -> Domain Registrar -> Authoritative nameservers.  After the recursive resolver queries the authoritative nameserver, it returns an IP address as an answer. 

### Recursion vs Iteration (nonrecursion)

Not every DNS resolution process has to follow these steps. There are two types of DNS queries:
- recursive - name for the resolution process used by a nameserver when it receives recursive queries.
- iterative - refers to the resolution process used by a nameserver when it receives iterative queries.

In **recursion**, a resolver sends a recursive query to a nameserver for information about a particular domain name. The queried nameserver is then obliged to respond with the requested data or with an error stating either that data of the requested type doesn’t exist or that the domain name specified doesn’t exist. The nameserver can’t just refer the querier to a different nameserver, because the query was recursive. 

A nameserver that receives a recursive query that it can’t answer itself will query the “closest known” nameservers. The closest known nameservers are the servers authoritative for the zone closest to the domain name being looked up - this is how the standard DNS lookup process occurs. In place of root nameservers we can also request different nameservers to inquire about the domain we wish to visit which  are more specific to the domain in question and cut the process short. 

In **iterative resolution**, a nameserver gives the best answer it already knows back to the querier and no additional querying is required. The queried nameserver consults its local data, looking for the data requested. If it doesn’t find the answer there, it finds the names and addresses of the nameservers closest to the domain name in the query in its local data and returns that as a referral to help the querier continue the resolution process. 

`dig +norec touchsupport.com`

**Choosing between authoritative nameservers** depends on a metric called *roundtrip time or RTT*.  Each time a BIND nameserver sends a query to a remote nameserver, it starts an internal stopwatch. When it receives a response, it stops the stopwatch and makes a note of how long that remote nameserver took to respond. When the nameserver must choose which of a group of authoritative nameservers to query, it simply chooses the one with the lowest roundtrip time.

### Caching:

A nameserver processing a recursive query may have to send out quite a few queries to find an answer, but it also discovers a lot of information about the domain namespace as it does so. Each time it’s referred to another list of nameservers, it learns that those nameservers are authoritative for some zone, and it learns the addresses of those servers. 

The nameserver even implements *negative caching*: if a nameserver responds to a query with an answer that says the domain name or data type in the query doesn’t exist, the local nameserver will also temporarily cache that information.

The next time a resolver queries the nameserver for data about a domain name the nameserver knows something about, the process is shortened quite a bit. The nameserver may have cached the answer, positive or negative, in which case it simply returns the answer to the resolver. Even if it doesn’t have the answer cached, it may have learned the identities of the nameservers that are authoritative for the zone the domain name is in and be able to query them directly.

### Time to Live - TTL
This also applies to negatively cached data: a nameserver must time out a negative answer after a period in case new data has been added on the authoritative nameservers.
